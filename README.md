# Verilog + Rust Hardware Lab

An educational interactive web app that teaches the full journey from logic gates to Rust firmware drivers.

**Live concept:** Build digital circuits visually → generate Verilog → simulate behavior → see how Rust firmware interacts with the hardware through memory-mapped registers.

---

## Quick Start

```bash
npm install
npm run dev
```

## What this app teaches

### The hardware/firmware mental model

Most software developers understand programs that run line-by-line. Hardware is different:

| Concept | Software (Rust) | Hardware (Verilog) |
|---|---|---|
| Execution | Line by line | All logic active simultaneously |
| Memory | Variables in RAM | Flip-flops and registers |
| State change | Assignment `=` | Rising clock edge |
| Time | CPU cycles | Wall clock, clock signal |
| Communication | Function calls | Wires and signals |

### What Verilog does

Verilog is a hardware description language, not a programming language. When you write:

```verilog
assign y = a & b;
```

You are describing a physical AND gate, not an instruction. The gate is always active. There is no "run this line" — the hardware evaluates it continuously.

Sequential logic (registers, counters, flip-flops) uses a clock:

```verilog
always @(posedge clk) begin
    if (reset) count <= 8'd0;
    else count <= count + 8'd1;
end
```

The `always @(posedge clk)` block only runs on the rising edge of the clock signal, capturing the new value into the register.

### What Rust firmware does

Rust firmware runs on a CPU (ARM Cortex-M, RISC-V, etc.) and talks to hardware through memory-mapped I/O:

```rust
// Hardware registers live at a specific address in memory
const PERIPH_BASE: usize = 0x4000_0000;

// Read STATUS register (volatile – hardware can change this anytime)
let status = unsafe { read_volatile(&(*regs).status) };

// Check if data is ready
if status & 0x1 != 0 {
    let data = unsafe { read_volatile(&(*regs).data) };
    // process data...
    // Clear interrupt
    unsafe { write_volatile(&mut (*regs).irq_clr, 1) };
}
```

**Why volatile?** The compiler optimizes by caching values. Without `volatile`, it might read STATUS once, decide it hasn't changed, and never read it again — missing hardware updates. `read_volatile` forces a fresh load from the actual hardware register on every call.

## How to use it

1. **Drag components** from the left toolbox onto the canvas.
2. **Connect components** by clicking an output port (right side of a node), then clicking an input port (left side of another node).
3. **Toggle input pins** by clicking on them (toggles 0/1).
4. **Step the clock** using the "Step" button to advance one rising edge.
5. **Run** lets the clock advance automatically at 2 Hz.
6. **View the Verilog** tab to see the generated hardware description.
7. **View the Rust** tab to see the generated firmware driver.
8. **View the Timeline** tab to see a waveform history of all signals.
9. **View the Firmware** tab to see the simulated firmware pipeline (when MMIO blocks are present).
10. **Load examples** from the dropdown to explore pre-built circuits.
11. **Tutorial** (top-right button) walks through 10 guided steps.
12. **Advanced** toggle opens a glossary of hardware concepts (FPGA, metastability, DMA, etc.).

### Controls

## Generated Verilog

The Verilog generator produces readable, commented output:

```verilog
// Generated by Verilog + Rust Hardware Lab
module counter8 (
    input wire clk,
    input wire reset,
    input wire enable,
    output wire count
);

    // Internal signals
    reg [7:0] Counter_count;

    // 8-bit Counter: increments on each clock edge when enabled
    always @(posedge clk) begin
        if (reset) begin
            Counter_count <= 8'd0;
        end else if (enable) begin
            Counter_count <= Counter_count + 8'd1;
        end
    end

    assign count = Counter_count;

endmodule
```

This is synthesizable Verilog — it could be processed by tools like Yosys + nextpnr to run on real FPGA hardware.

## Generated Rust

The Rust generator produces a memory-mapped driver skeleton:

```rust
#[repr(C)]
pub struct UartRxRegs {
    /// bit 0 = RX_READY, bit 1 = OVERRUN  [ro]  offset: +0
    pub status: u32,
    /// Received byte (read to clear RX_READY)  [ro]  offset: +4
    pub data: u32,
    /// bit 0 = RX_ENABLE, bit 1 = IRQ_ENABLE  [rw]  offset: +8
    pub control: u32,
    /// Write 1 to clear RX interrupt  [wo]  offset: +12
    pub irq_clr: u32,
}

pub struct UartRx {
    regs: *mut UartRxRegs,
}

impl UartRx {
    pub const unsafe fn new(base: usize) -> Self {
        Self { regs: base as *mut UartRxRegs }
    }

    pub fn rx_ready(&self) -> bool {
        self.status() & 0x1 != 0
    }

    pub fn read_data(&self) -> u8 {
        self.data() as u8
    }

    pub fn clear_irq(&self) {
        self.set_irq_clr(1);
    }
}
```

## firmbuf-rs Integration

The firmbuf-rs pipeline example demonstrates the standard embedded pattern for high-throughput peripheral data:

```
Verilog peripheral
    ↓  raises IRQ, sets STATUS.RX_READY, loads DATA register
Rust interrupt handler / polling loop
    ↓  read_volatile(STATUS) → read_volatile(DATA) → clear IRQ
firmbuf-rs RingBuf (lock-free SPSC)
    ↓  producer pushes bytes (ISR side)
    ↓  consumer pops bytes (main loop side)
Packet parser (state machine)
    ↓  assembles framed packets from byte stream
Application logic
```

The ring buffer decouples the ISR (which must be fast) from the parser (which may be slower). This pattern appears in virtually every embedded peripheral driver: UART, SPI, I2C, USB CDC, BLE, CAN.

## Architecture

```
src/
├── simulator/       Pure TypeScript simulation engine (no React)
│   ├── types.ts     Core data model: CircuitNode, Wire, Port, SignalValue
│   ├── circuit.ts   Topological sort, combinational eval, clock stepping
│   ├── gates.ts     NOT/AND/OR/XOR/Comparator/Mux evaluators
│   ├── sequential.ts DFF/Register8/Counter8 clocked state updates
│   ├── memoryMapped.ts MMIO register block simulation
│   ├── timeline.ts  Signal recording for waveform display
│   └── examples.ts  5 built-in example circuits
├── generators/      Code generators (pure TypeScript)
│   ├── verilog.ts   Circuit → Verilog text
│   ├── rust.ts      Circuit → Rust driver skeleton
│   └── explanations.ts Per-component educational text
├── firmware/        Firmware pipeline models
│   ├── ringBufferModel.ts     SPSC ring buffer simulation
│   ├── packetParserModel.ts   Framed packet parser state machine
│   └── rustIntegrationModel.ts Full pipeline orchestration
├── store/
│   └── circuitStore.ts  Zustand global state
├── content/
│   ├── tutorials.ts  10-step guided tutorial
│   └── glossary.ts   Advanced concepts glossary
└── components/      React UI (reads from store, renders circuit)
```

**Design principles:**
- Simulator is pure functions with no React dependency
- Generators take a Circuit snapshot and return strings
- Zustand store calls simulator functions and stores results
- Components only render, never compute

## Limitations

This is an educational simulator, not a professional HDL tool:

- No timing analysis (propagation delays are ignored)
- Single clock domain only
- Fixed set of 13 supported components
- Generated Verilog does not pass through synthesis verification
- Bus signals simplified to 8-bit
- Wire routing is Manhattan (no auto-routing)
- No X-state propagation for bus signals
